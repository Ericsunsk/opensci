# 项目阅读说明

## 项目定位

这个项目被定义为一个 `评审工作台`，而不是一个泛化的项目发现页或抽象评分面板。

它关注的是科研支持决策里最容易失真的一段流程：

- 项目刚进入评审池时，证据往往分散且不完整
- 真实支持动作常常发生在验证尚未充分完成的阶段
- 一次性承诺会把大量下行风险前置
- 支持之后如果没有持续记录，前序判断很快就失去可追踪性

因此，这个工作台并不试图覆盖完整平台，而是聚焦在四个最核心的工作对象：

1. `案件`
   谁提交了什么请求，为什么会进入本轮评审。
2. `证据`
   每条证据来自哪里、由谁提交、由谁验证、最近一次何时复查。
3. `动作`
   补证、风险升级、观察名单与 tranche 批准都需要留下上下文。
4. `决策包`
   最终输出必须是委员会可以复核的支持方案，而不是页面内部分散的判断片段。

## 为什么要从“展示页”收敛到“工作台”

科研支持产品很容易陷入两个常见误区：

- 把大量解释性内容堆在首页，导致用户先理解理念，再开始工作
- 把评审过程做成一个调参和打分的沙盘，最终没有可提交的结果对象

这次收敛的目标，就是把产品默认入口改成真实工作流：

- `Inbox`
  先判断今天该处理哪个案件
- `Case Summary`
  快速看清提交背景、申请内容与当前阶段
- `Evidence & Review`
  处理证据来源链、补证请求与评审动作
- `Decision Packet`
  生成可进入委员会队列的支持方案

## 当前版本解决了什么

- 让评审入口从“概念介绍”变成“案件处理”
- 让证据展示从“信息堆砌”变成“来源链”
- 让高风险动作从“一键改状态”变成“带原因、责任人和时间点的操作”
- 让最终输出从“调参结果”变成“决策包”

## 当前版本仍然刻意没有展开的范围

- 完整的钱包与 token 流程
- 链上结算与合约细节
- 自动化科研执行系统
- 完整的 contributor identity / reputation graph

这些方向仍然属于同一产品体系，但更适合在评审层被证明有价值之后继续扩展。
