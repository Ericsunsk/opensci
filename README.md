# OPENSCI 项目支持决策台

一个面向科研项目评审与支持决策的单页工作台原型。它把项目判断拆成 `证据审查 -> 补证请求 -> 风险标记 -> tranche 决策 -> 后续监控` 五个连续动作，而不是停留在一次性的“支不支持”。

- 在线预览：<https://ericsunsk.github.io/opensci/>
- 评审说明：[`docs/reviewer-brief.md`](./docs/reviewer-brief.md)
- 演示语言：默认中文，支持中 / 英切换

如果你想先快速理解这个原型为什么贴合 OPENSCI、它解决的核心问题是什么、以及演示时该怎么讲，可以先看 [docs/reviewer-brief.md](./docs/reviewer-brief.md)。

这个产品原型希望帮助用户完成三件事：

1. 判断一个项目是否值得支持。
2. 决定应该如何支持，而不只是“支不支持”。
3. 在初次决策之后，继续补齐证据、记录评审动作，并持续接收信任更新。

## 为什么做这个产品方向

这个岗位并不是一个泛泛的 AI PM 岗。JD 明确强调了科研社区、贡献证明、声誉体系、治理机制、生态参与和重点项目推进。与此同时，OPENSCI 公开材料目前也强调了里程碑资助、科研协作、贡献归因和 AI 辅助科研基础设施。这个原型刻意对齐的是这些公开叙事里与评审最相关的一层：

- 它本质上是一个 `信任 + 治理 + 决策支持` 问题。
- 它借用了 OPENSCI 公开产品叙事里的几个关键点：里程碑资助、可检查证据、贡献归因、支持后的信任更新。
- 它体现的是“如何在不确定性下设计更稳健的产品结构”，这比一个泛泛的 AI Agent 工作流更接近 OPENSCI 的真实业务环境。

更具体地说，这个原型聚焦在 `资助 / 归因 / 协作` 三者之间最容易失真的一层：

- `资助`
  支持资源不是一次性给出，而是按里程碑分阶段释放。
- `归因`
  证据和贡献记录比叙事热度更重要。
- `协作`
  决策不是批准就结束，支持开始之后仍然要持续更新信任。

## 产品概念

`Decision Console` 是一个面向评审者的产品，用来在投入金钱、时间或参与席位之前，评估一个高不确定性的科研项目。

它建立在四个原则上：

- `证据先于叙事`
  不是所有信息都应该被同等对待，可检查证据和外部校验应当拥有更高权重。
- `支持分阶段释放`
  好的系统不只是判断项目“好不好”，而是改变投入结构，让用户可以做条件支持，而不是一次性 all-in。
- `持续信任更新`
  对于长周期科研结果，产品必须在支持后继续观察，而不是只在最开始给一次分数。
- `显式暴露失败模式`
  产品应该告诉用户：这个判断为什么可能错、最可能从哪里失效。

## 原型结构

当前原型是一个单页评审工作台，分成四层：

1. `项目队列`
   用户从一组待支持项目开始，先看到项目的信任、证据、异常风险和当前评审状态。
2. `评审工作区`
   选中项目后，工作台会显示：
   - 当前评审阶段
   - 评审负责人
   - 下一个决策节点
   - 仍未完成的补证请求
   - 发起补证、标记风险、批准 tranche、移入观察名单等动作
3. `项目详情区`
   每个项目会展示：
   - 用户被要求相信的核心论点
   - 加权后的证据，而不是简单信息堆砌
   - 里程碑信心和分阶段释放条件
   - 异常雷达和潜在夸大 / 造假风险
   - 评审备注和审计记录
   - 支持后的持续反馈
4. `决策工作台`
   用户可以设定支持方式、支持规模、风险偏好，以及是否启用分阶段释放、第三方复现和托管机制，系统再输出建议和原因。

## 重新定义后的核心问题

这个原型解决的不是泛泛的“发现好项目”，而是一个更窄、更实际的评审问题：

`如何帮助评审者在证据不完整的情况下，对科研项目做出更克制、更可追踪的支持决策，同时让归因、治理和支持后的信任更新始终可见？`

这个重构很重要，因为真正的失败并不只是“坏项目存在”，而是：

- 证据还不够强时，支持已经先发生了
- 叙事势能跑在可检查证明前面
- 一次性承诺太重，导致下行风险全部前置

## 六步演示叙事

这个 demo 最适合这样讲：

1. `项目进入队列`
   评审者看到一个看起来有潜力的科研项目，但证据分散、验证周期很长。
2. `论点变得可读`
   系统先把项目的核心主张讲清楚，再把已验证、部分验证和缺失证据分开呈现。
3. `真正的风险被暴露`
   风险不只是项目会不会失败，而是用户可能在复现、归因或里程碑条件还不够强时就先做了承诺。
4. `评审者开始行动`
   缺失材料可以被请求，风险可以被标记，项目也可以重新送回治理复核。
5. `支持结构被设计出来`
   支持不再是 yes / no，而是被转成基于 tranche 的阶段性释放方案。
6. `信任持续更新`
   备注、审计记录和支持后的项目更新，让这个项目在初始决策后仍然保持可追踪。

## 核心机制

### 1. 证据加权模型

这个产品不会把所有输入一视同仁。公开原始数据、独立复现和团队过往交付记录，应当比叙事热度更重要。

为什么重要：

- 高不确定性项目常常输在信息不对称
- 如果不做证据加权，用户很容易高估故事和社会认同

### 2. 里程碑分段支持

系统支持“条件性支持”。用户不需要做一次性不可逆投入，而是可以把后续支持绑定在里程碑证明上。

为什么重要：

- 如果项目没达到检查点，可以明显降低下行风险
- 它也会反过来约束团队更诚实地汇报进展

### 3. 异常雷达

系统会显式列出脆弱点，例如：原始日志缺失、复现不足、审批过于中心化、支持请求无约束、叙事热度超过证据成熟度。

为什么重要：

- 它让失败模式变得可检查
- 也让评审者知道造假或夸大最可能先从哪里暴露

### 4. 持续反馈回路

产品不会在“做出决策”之后结束。它会继续追踪项目更新，让信任随着后续行为上升或衰减。

为什么重要：

- 长周期科研结果很难即时验证
- 用户真正需要的是支持后的监控回路，而不只是一次性评分

## 操作化模块

### 1. 评审流程

工作台会明确显示项目当前处于哪一步：入池、证据审查、治理复核、tranche 决策或持续监控。

为什么重要：

- 它把抽象判断变成了真实流程
- 评审者可以立刻看到堵点、关口和责任归属

### 2. 动作面板

评审者可以在同一屏里直接发起动作：请求更多证据、标记风险、批准 tranche、移入观察名单。

为什么重要：

- 产品更像工作工具，而不只是展示页面
- 每个动作都会改变状态，并写入审计记录

### 3. 补证面板

缺失证明不会只被模糊地写成“不确定性”，而是会变成带责任人、截止日期和状态的待办事项。

为什么重要：

- 不确定性变成了可推进的工作
- 评审者可以明确知道下一个决策关口被什么阻塞

### 4. 评审备注与审计轨迹

当前判断依据会进入评审备注，所有关键动作都会写入审计日志。

为什么重要：

- 后续评审者可以理解为什么判断姿态发生变化
- 治理负责人和支持运营也可以事后回看整个过程

## 设计取舍与失效分析

### 1. 如果项目在造假，系统最可能从哪里失效？

最脆弱的地方，永远是那些没有建立在可检查证据之上的信号。实际中，系统通常会在这些地方失效：

- 伪造或低质量证据被当成已验证证据
- 叙事势能的权重高于原始证明
- 团队在没有里程碑证明前就能拿到无约束支持
- 缺少第三方验证或复现要求

这也是为什么原型里会把 `trust score`、`evidence coverage` 和 `anomaly risk` 分开呈现，并允许用户加上复现和托管保护。

### 2. 如果用户频繁做错决策，问题通常在哪里？

通常会出在四个地方：

- 证据模型本身错了：高价值信号被低估，低质量信号被高估
- UI 把不确定性藏起来了：用户看到了推荐，但没理解前提和 caveat
- 支持结构太二元：只有“支持 / 不支持”，没有“条件支持”
- 反馈回路太弱：系统没有学习哪些早期信号真正预测了项目结果

在真实产品里，我会用推荐准确率、支持后悔率、里程碑失守率和错误高置信率来闭环校准。

### 3. 最关键的机制是什么？

最关键的是 `里程碑分段支持`。

原因很简单：

- 不确定性无法被彻底消灭
- 造假也很难被百分之百提前识别
- 但产品仍然可以通过改变投入结构来显著降低下行风险

这比假装系统一开始就能完美判断项目优劣，更现实。

### 4. 我故意放弃了哪些信息？为什么？

我刻意没有放进来这些内容：

- 面向用户完全展开的详细评分公式
- 点赞量、粉丝量等社交热度指标
- 会压垮非专家评审者的过深技术细节
- 过早展开的链上实现细节

原因是：

- 过度暴露评分逻辑会提高被 game 的风险
- 热度非常容易被操纵，而且通常和科研可信度弱相关
- 用户目标是做出更好决策，而不是成为领域专家
- 产品应先把信任结构和支持结构做对，再谈实现复杂度
